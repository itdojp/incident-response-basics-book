# 第2章：情報収集（変更点/影響範囲/再現条件/証跡）

## この章で学ぶこと

- 直近変更点（デプロイ/設定/依存）を確認する
- 影響範囲を特定し、優先順位を決める
- 証跡（ログ/設定/メトリクス）を保全する

## 成果物（または判断基準）

- 状況共有（付録: [状況共有](../../appendices/templates/status-update/)）
- 初動チェック（付録: [チェックリスト](../../appendices/checklists/)）

## 本文

情報収集は“あとで確認できる形”で行う。口頭で集めた情報は抜けやすく、判断の根拠が残らない。

### 最低限の収集項目

- 直近の変更（いつ/何を/誰が）
- 影響範囲（利用者、機能、期間）
- 再現条件（入力、負荷、時間帯）
- 証跡（ログ、メトリクス、設定差分）

収集した情報は“状況共有テンプレ”に落とし込み、関係者が同じ前提で動けるようにする。

## 具体例（場当たり→再現性）

### 悪い例（場当たり）

```md
変更点: たぶん何か変えた
影響範囲: よく分からない
証跡: あとで見ればいい
```

### 良い例（再現性）

```md
変更点: 10:05 に v1.2.3 をデプロイ（PR#123）
影響範囲: 決済APIのみ、10:12〜
再現: 特定の入力でタイムアウト
証跡: 期間のログ/メトリクス/設定を保存
共有: 状況共有テンプレで周知
```

## チェックリスト

- [ ] 直近変更点が特定できている
- [ ] 影響範囲が説明できる
- [ ] 証跡が保全されている
- [ ] 状況共有が更新されている

## まとめ

- 調査は“仮説→検証→記録”をセットで回す
- インシデント対応は“技術”だけでなく“運用（連絡/判断/役割）”が重要
