# 第6章：連絡とエスカレーション（判断基準/情報粒度）

## この章で学ぶこと

- 連絡は“頻度と粒度”を固定する
- エスカレーション基準を事前に持つ
- 必要支援を具体化する

## 成果物（または判断基準）

- 状況共有（付録: [状況共有](../../appendices/templates/status-update/)）
- エスカレーション（付録: [エスカレーション](../../appendices/templates/escalation/)）

## 本文

連絡の失敗は、情報が多すぎる/少なすぎる、更新がない、のいずれか。テンプレで固定する。

### 共有の最小セット

- いま何が起きているか
- 何を試したか（結果）
- 次に何をするか
- 影響見込み

エスカレーションは“判断根拠”と“必要支援”が揃っていると速い。

## 具体例（場当たり→再現性）

### 悪い例（場当たり）

```md
原因調査中です（以上）
次に何をするかが不明
```

### 良い例（再現性）

```md
状況: 決済APIで5xx増加
試行: 直近変更を比較→変更Aが疑わしい
次: ロールバック可否確認→実施判断
支援: DB担当に接続状況の確認依頼
```

## チェックリスト

- [ ] 共有が定期的
- [ ] 次アクションが明確
- [ ] エスカレーションの判断根拠がある

## まとめ

- 調査は“仮説→検証→記録”をセットで回す
- インシデント対応は“技術”だけでなく“運用（連絡/判断/役割）”が重要
