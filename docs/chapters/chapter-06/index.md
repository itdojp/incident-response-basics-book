# 第6章：連絡とエスカレーション（判断基準/情報粒度）

## この章で学ぶこと

- 連絡は“頻度と粒度”を固定する
- エスカレーション基準を事前に持つ
- 必要支援を具体化する

## 成果物（または判断基準）

- 状況共有（付録: [状況共有](../../appendices/templates/status-update/)）
- エスカレーション（付録: [エスカレーション](../../appendices/templates/escalation/)）

## 本文

連絡の失敗は、情報が多すぎる/少なすぎる、更新がない、のいずれか。テンプレで固定する。

### 共有の最小セット

- いま何が起きているか
- 何を試したか（結果）
- 次に何をするか
- 影響見込み

### 更新頻度の目安（Severity別）

最小の目安です。組織の窓口/顧客影響/法令要件がある場合はそれを優先する。

| Severity | 更新頻度（目安） | 共有先（例） | 粒度（最小） |
|---|---|---|---|
| P0 | 10〜15分ごと | 当番/関係者/責任者/CS | 影響、暫定判断、次アクション、ETA、リスク |
| P1 | 30分ごと | 関係者/CS | 影響、進捗、次アクション、判断ポイント |
| P2 | 60〜120分 or 節目 | 関係者 | 節目（原因候補/暫定対応/復旧見込み）で更新 |
| P3 | 節目のみ | 関係者 | 予定化（いつ直す/どう検証するか）を明確化 |

エスカレーションは“判断根拠”と“必要支援”が揃っていると速い。

## 具体例（場当たり→再現性）

### 悪い例（場当たり）

```md
原因調査中です（以上）
次に何をするかが不明
```

### 良い例（再現性）

```md
状況: 決済APIで5xx増加
試行: 直近変更を比較→変更Aが疑わしい
次: ロールバック可否確認→実施判断
支援: DB担当に接続状況の確認依頼
```

## チェックリスト

- [ ] 共有が定期的
- [ ] 次アクションが明確
- [ ] エスカレーションの判断根拠がある

## まとめ

- 調査は“仮説→検証→記録”をセットで回す
- インシデント対応は“技術”だけでなく“運用（連絡/判断/役割）”が重要
