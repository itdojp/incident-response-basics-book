# 第8章：ポストモーテム（真因/再発防止/検知・手順改善）

## この章で学ぶこと

- 真因を“仕組み”として説明する
- 再発防止を“行動（期限/担当/検証）”に落とす
- 検知と手順（Runbook）に反映する

## 成果物（または判断基準）

- ポストモーテム（付録: [ポストモーテム](../../appendices/templates/postmortem/)）
- フォローアップ管理（アクション一覧）

## 本文

ポストモーテムは反省会ではない。再発防止が行動に落ち、追跡できることが目的。

### 最低限の構造

- 事象要約
- 真因（直接原因/背後要因）
- 再発防止（期限/担当/検証）
- 検知改善
- 手順改善（Runbook更新）

## 具体例（場当たり→再現性）

### 悪い例（場当たり）

```md
原因: 気をつける
対策: 注意する
フォローアップがない
```

### 良い例（再現性）

```md
真因: 接続プール設定が負荷変動に耐えない
対策: 設定見直し + 負荷試験（期限/担当）
検知: 閾値をSLIベースに再設計
手順: Runbookに初動とロールバックを追記
フォローアップ: アクション一覧で追跡
```

## チェックリスト

## まとめ

- 調査は“仮説→検証→記録”をセットで回す
- インシデント対応は“技術”だけでなく“運用（連絡/判断/役割）”が重要
